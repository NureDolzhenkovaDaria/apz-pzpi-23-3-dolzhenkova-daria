3 ОПИС ВИКОНАНОЇ РОБОТИ

3.1 Загальні відомості про систему
Назва: Canva.
Розробник: Canva Pty Ltd (Австралія).
Призначення: хмарна SaaS-платформа для онлайн-графічного дизайну.
Область застосування: Web, Mobile, Cloud.
Користувачі: SMM, маркетологи, студенти, викладачі, малий бізнес.
Основні функції: WYSIWYG-редактор, спільна робота (колаборація), готові шаблони/медіа, ШІ-інструменти, експорт у PDF/PNG/MP4.
3.2 Архітектурний стиль системи
Платформа комбінує три архітектурні стилі:
    Товстий клієнт (Thick Client): логіка інтерфейсу та рендерингу виконується у браузері для мінімізації затримок.
    Мікросервіси (Microservices): бекенд розбитий на автономні сервіси (авторизація, рендеринг, медіа, підписки) для стабільності платформи.
    Подієво-орієнтований (Event-Driven): важкі завдання (конвертація відео, експорт у PDF) обробляються асинхронно через черги повідомлень.
Обґрунтування вибору: мікросервіси забезпечують незалежне масштабування та автономність компонентів (збій одного сервісу не ламає всю систему). Клієнтський рендеринг гарантує плавність інтерфейсу (60 FPS) при взаємодії з елементами.
3.3 Основні компоненти системи
Архітектурна структура Canva розділена на чотири основні рівні:
    Інтерфейс користувача (Front-end):
    React + TypeScript: побудова WYSIWYG-редактора; строга типізація об'єктів дизайну (координати, шари).
    MobX: менеджмент стану додатка та миттєве оновлення інтерфейсу в реальному часі.
    HTML5 Canvas / SVG: базовий фундамент для малювання та маніпуляцій з графікою.
    Серверна частина та рівень бізнес-логіки (Back-end):
    Jetty: легковагий вебсервер для розгортання додатків.
    Java + Spring Boot (Spring MVC): реалізація мікросервісів, REST API, авторизації та бізнес-логіки.
    RxJava: реактивне асинхронне керування потоками даних при спільній роботі користувачів.
    Рівень зберігання даних (Data Layer):
    Hibernate (ORM): мапінг Java-моделей у реляційну БД для роботи з профілями та проєктами.
    Amazon S3: хмарне об'єктне сховище для важкого контенту (зображення, шрифти, відео).
    Допоміжні сервіси та мобільні платформи:
    Мобільні клієнти: нативні додатки на iOS (Swift, Objective-C) та Android (Kotlin).
    Сервіси рендерингу: фонові сервери для конвертації JSON-опису дизайну у PNG, JPG або PDF.
3.4 Взаємодія компонентів
Схема архітектурної структури та взаємодії наведена у Додатку Б (див. Рисунок Б.1). Обмін даними побудовано на трьох механізмах:
    REST API (HTTP/HTTPS): синхронні запити для базових операцій (автентифікація, завантаження шаблонів, пошук медіа, зміна профілю).
    WebSockets: двосторонній зв'язок для колаборації в реальному часі. Зміна стану елементів у MobX відправляється на сервер, а RxJava миттєво транслює її іншим учасникам сесії.
    Асинхронний експорт: при запиті на завантаження важких файлів (PDF/MP4) сервер повертає статус 202 Accepted і ставить задачу в чергу. Після фонового рендерингу клієнт отримує посилання на готовий файл в Amazon S3.
3.5 Масштабованість та надійність
Для забезпечення високої доступності та швидкої обробки графіки в Canva реалізовано такі механізми:
    Балансування та хмара: розгортання в Amazon EC2 з використанням автоматичного масштабування для динамічного виділення серверів під поточне навантаження.
    Відмовостійкість: асинхронний підхід на базі RxJava. Важкі завдання (рендеринг, експорт відео/PDF) виносяться у фонові модулі, що запобігає блокуванню потоків основного API.
    Кешування: зниження навантаження на реляційні БД шляхом кешування сесій користувачів, статичних шаблонів та шрифтів на проміжних шарах сховища (Redis).
    Надійність збереження контенту: ізольоване зберігання медіафайлів в об'єктному сховищі Amazon S3 з вбудованою реплікацією та гарантією доступності даних на рівні 99.99%.
3.6 Приклади програмної реалізації
Нижче наведено приклад промту до ШІ та коду на Java (Spring Boot + Hibernate), який демонструє архітектурний принцип обробки HTTP-запиту від клієнта та збереження метаданих дизайну в базу даних.
Запит до сервісу ШІ (Prompt):
Напиши простий код для розуміння: Java Spring Boot контролер та Hibernate сутність (entity) для збереження метаданих дизайну (id, title, user_id). 
Фрагмент програмного коду:
1  package ua.canva;
2  
3  import jakarta.persistence.*;
4  import org.springframework.web.bind.annotation.*;
5  
6  @Entity
7  @Table(name = "designs")
8  class Design {
9      @Id @GeneratedValue(strategy = GenerationType.IDENTITY) 
10     public Long id;
11     public String title;
12     public String userId;
13 }
14 
15 @RestController
16 @RequestMapping("/api/designs")
17 public class DesignController {
18 
19     @PostMapping("/save")
20     public String save(@RequestBody Design design) {
21         if (design.userId == null) return "Error: Invalid User";
22         // Імітація роботи Hibernate ORM та збереження в БД
23         return "Saved successfully with ID: " + design.id;
24     }
25 }
